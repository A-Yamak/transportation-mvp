Project Overview

Al-Sabiqoon is a modern full-stack web application built with Laravel 12 (backend API) and Vue 3 (frontend SPA). The project follows industry best practices with Docker containerization, Kubernetes-ready deployment, and a comprehensive testing strategy.

Backend API

Laravel 12 with PHP 8.4, serving RESTful APIs with OAuth2 authentication via Passport.

Frontend SPA

Vue 3 with TypeScript, featuring a glassy light/dark theme built with Tailwind CSS 4.

Infrastructure

Docker Compose for local development, Kubernetes manifests for production deployment.

Domain

alsabiqoon.com (frontend) and api.alsabiqoon.com (backend API).

System Architecture

How the pieces fit together

The application follows a decoupled architecture where the frontend and backend are completely separate applications that communicate via HTTP APIs. 

                           +-------------------+
     User Browser          |    Frontend       |
          |                |    (Vue 3)        |
          |                |    Port 5173      |
          v                +-------------------+
    +------------+                |
    |   Nginx    |                | HTTP API Calls
    |   (Prod)   |                v
    +------------+         +-------------------+
          |                |    Backend        |
          |                |    (Laravel)      |
          v                |    Port 8000      |
    +------------+         +-------------------+
    |  Frontend  |                |
    |  Static    |         +------+------+
    |  Files     |         |             |
    +------------+    +--------+    +--------+
                      | MySQL  |    | Redis  |
                      | 3307   |    | 6379   |
                      +--------+    +--------+

Request Flow

  1. User opens browser → Loads Vue 3 frontend from port 5173 (dev) or Nginx (prod)
  2. Frontend makes API call → Sends HTTP request to backend at port 8000
  3. Backend processes request → Laravel handles authentication, business logic, database queries
  4. Backend returns JSON → Frontend receives data and updates the UI

Key Concepts Explained

If you're new to these technologies, here's what each one does and why we use it:

Laravel Octane

What it is: Octane supercharges Laravel by keeping your application in memory between requests, rather than booting it fresh for every request.

Why it matters: Traditional PHP loads your entire application for each HTTP request, then throws it away. Octane boots once and handles thousands of requests, making your app 2-10x faster.

Simple analogy: Instead of starting your car engine for each trip to the mailbox, you keep it running all day.

FrankenPHP

What it is: A modern PHP application server written in Go. It's one of the "engines" that can power Laravel Octane (alternatives: Swoole, RoadRunner).

Why we chose it: FrankenPHP is the newest option, supports HTTP/3, automatic HTTPS, and integrates well with Docker. It's created by the Symfony team.

Laravel Horizon

What it is: A beautiful dashboard and queue manager for Laravel's Redis-powered queues.

What are queues? Some tasks are too slow to do during a web request (sending emails, processing images, generating reports). Queues let you "queue" these jobs to run in the background.

What Horizon does: It monitors your queues, shows you what's running, what failed, how long jobs take, and lets you retry failed jobs. 

// Example: Instead of sending email during request (slow)
Mail::to($user)->send(new WelcomeEmail());

// Queue it for background processing (fast response to user)
Mail::to($user)->queue(new WelcomeEmail());

Supervisor

What it is: A process manager that keeps your background processes running. If they crash, Supervisor restarts them automatically.

What we use it for:

  • Running php artisan horizon (queue worker)
  • Running php artisan schedule:work (task scheduler)

Simple analogy: It's like a babysitter for your background processes - if they fall asleep, it wakes them back up.

Laravel Tinker

What it is: An interactive command-line shell (REPL) for Laravel. You can type PHP code and see results immediately.

Why it's useful: Perfect for testing code, querying the database, or experimenting without creating test files. 

# Start Tinker
make tinker

# Inside Tinker, you can:
>>> User::count()
=> 42

>>> User::find(1)->name
=> "John Doe"

>>> User::create(['name' => 'Jane', 'email' => 'jane@example.com'])

PHPUnit

What it is: The standard testing framework for PHP. It lets you write automated tests to verify your code works correctly.

Why testing matters: Tests catch bugs before users do. When you change code, run tests to make sure you didn't break anything. 

// Example test: tests/Feature/UserTest.php
public function test_user_can_register(): void
{
    $response = $this->post('/api/register', [
        'name' => 'John Doe',
        'email' => 'john@example.com',
        'password' => 'password123',
    ]);

    $response->assertStatus(201);
    $this->assertDatabaseHas('users', ['email' => 'john@example.com']);
}
# Run all tests
make test

# Run specific test
make test-filter name="UserTest"

Redis

What it is: An in-memory data store. Think of it as a super-fast database that stores data in RAM instead of disk.

What we use it for:

  • Cache: Store frequently-accessed data (user sessions, API responses) for faster retrieval
  • Sessions: Store user login sessions
  • Queues: Store background jobs waiting to be processed

Redis Isolation (k3s): When sharing Redis across environments, use different database numbers:

EnvironmentDB NumbersPrefix
Local0, 1, 2alsabiqoon_local_
Staging3, 4, 5alsabiqoon_staging_
Production6, 7, 8alsabiqoon_prod_

Telescope & Horizon Authorization

What they are: Laravel debug dashboards. Telescope shows requests, queries, logs, exceptions. Horizon monitors Redis queues and background jobs.

Authorization: In non-local environments, access is restricted via email whitelist: 

DASHBOARD_AUTHORIZED_EMAILS=admin@alsabiqoon.com,dev@alsabiqoon.com

Telescope Recording Modes:

  • all - Records everything (Local, Staging)
  • errors_only - Only exceptions, failed requests/jobs, logs (Production)
  • important_only - Critical errors only (High-traffic production)

Docker & Docker Compose

What Docker is: Docker packages your application with all its dependencies into "containers" - isolated environments that run the same everywhere.

Why it matters: "It works on my machine" becomes "It works on every machine." No more installing PHP, MySQL, Redis separately - Docker handles everything.

What Docker Compose is: Defines and runs multiple containers together. Our compose.yaml defines backend, frontend, MySQL, Redis, etc. 

# Start all services (creates containers, networks, volumes)
make up

# Stop all services
make down

# View what's running
make ps

Kubernetes (K8s)

What it is: A platform for running containerized applications in production at scale. It handles deployment, scaling, load balancing, and self-healing.

Docker vs Kubernetes:

  • Docker: Builds and runs containers on one machine
  • Kubernetes: Orchestrates containers across many machines in production

When you use it: For production deployment. Locally, Docker Compose is sufficient.

Laravel Passport (OAuth2)

What it is: Full OAuth2 server implementation for Laravel. It handles API authentication with access tokens.

How it works:

  1. User logs in with email/password
  2. Server validates credentials and returns an "access token"
  3. Frontend stores the token and includes it in every API request
  4. Server validates the token to identify the user
// Frontend sends token with every request
fetch('/api/user', {
    headers: {
        'Authorization': 'Bearer eyJ0eXAiOiJKV1QiLCJhbGc...'
    }
})

Development Workflow

Daily Development

  1. Start your day: make up to start all services
  2. Write code: Changes auto-reload (backend via Octane, frontend via Vite HMR)
  3. Test: make test to run tests
  4. End your day: make down to stop services

Adding a New Feature

  1. Create a new branch: git checkout -b feature/my-feature
  2. Write tests first (TDD approach)
  3. Implement the feature
  4. Run tests: make test
  5. Lint code: make lint
  6. Commit and push

Database Changes

  1. Create migration: make artisan cmd="make:migration create_posts_table"
  2. Edit the migration file in backend/database/migrations/
  3. Run migration: make migrate
  4. If you made a mistake: make fresh resets everything
Warning: make fresh destroys all data!

Only use make fresh in development. It drops all tables and recreates them.
